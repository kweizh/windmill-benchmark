# Deploy a Cron Schedule Asset with the Windmill CLI ## Background Windmill is an open-source workflow engine. Workspaces are managed code-first via the `wmill` CLI: scripts live as `f/<folder>/<name>.py` (or `.ts`, etc.) and schedules live as `*.schedule.yaml` files. `wmill sync push --yes` deploys the whole tree to the workspace. You will author a minimal Python heartbeat script, declare a cron schedule asset that runs it every minute at second 0 in `Asia/Shanghai`, and push both to the cloud Windmill workspace at REDACTED. ## Requirements * Author a minimal Python script with `def main(): return {"alive": True}`. * Author a schedule asset (Windmill `*.schedule.yaml` file) that triggers the heartbeat script every minute at second 0, in the `Asia/Shanghai` timezone, and is enabled. * Configure `wmill.yaml` so a single invocation of `wmill sync push --yes` deploys both the script and the schedule to the cloud Windmill workspace. * Use the cloud Windmill instance `REDACTED` (do NOT try to start a local Windmill server). ## Implementation Hints * The `wmill` CLI is preinstalled. Authenticate non-interactively against the cloud instance using the `WINDMILL_TOKEN` and `WINDMILL_WORKSPACE` environment variables (e.g. `wmill workspace add ... --token "$WINDMILL_TOKEN" REDACTED/`). The workspace ID equals `WINDMILL_WORKSPACE`. * Read the current run id from the `ZEALT_RUN_ID` environment variable and append it to every Windmill path you create, so concurrent runs do not collide. Use the folder `f/eval_${ZEALT_RUN_ID}` for all assets in this task. * Use Windmill's documented cron syntax (croner) for "every minute at second 0". The seconds field is the first field. Document: https://www.windmill.dev/docs/core_concepts/scheduling . * A Windmill schedule file is a `*.schedule.yaml` whose path on disk maps directly to its workspace path (the filename without the `.schedule.yaml` suffix). Required keys are `schedule`, `timezone`, `enabled`, `script_path` (the path of the runnable to trigger), and `is_flow` (`false` for a plain script). Documented at https://www.windmill.dev/docs/core_concepts/scheduling and https://github.com/windmill-labs/windmill/tree/main/windmill-yaml-validator . * Schedules are skipped by default by `wmill sync push`. Enable them in `wmill.yaml` via `includeSchedules: true`, and make sure your `includes:` glob covers the folder that contains your script and schedule. * After `wmill sync push --yes` succeeds, write a short log line for each deployed asset to the log file under the project so the verifier can locate them. ## Acceptance Criteria * Project path: /home/user/myproject * Ensure the real `wmill sync push --yes` action is executed against `REDACTED` and the workspace contains the deployed assets afterwards. * Log file: /home/user/myproject/output.log * In the log file, print two lines in this exact format (no extra whitespace inside the colons): * `Script path: <script_path>` * `Schedule path: <schedule_path>` Both `<script_path>` and `<schedule_path>` MUST start with `f/eval_${ZEALT_RUN_ID}/`. * The deployed script asset must exist in the workspace and must be authored in Python (language reported by the Windmill API as `python3`). * The deployed schedule asset must satisfy ALL of: * Its cron expression literal equals the croner 6-field form for "every minute at second 0": `0 * * * * *`. * Its `timezone` field equals `Asia/Shanghai`. * Its `script_path` field equals the deployed Python script's path (the same `f/eval_${ZEALT_RUN_ID}/...` path that was written to the log as `Script path:`). * Its `enabled` field equals `true`. * It targets a script (not a flow), i.e. `is_flow` is `false` on the schedule.
