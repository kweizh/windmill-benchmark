# Windmill WAC: Multi-Stage Deployment Pipeline with Suspend / Resume

## Background

You have an authenticated [Windmill](https://www.windmill.dev/) workspace on the cloud instance at `https://app.windmill.dev`. A Bearer API token and the target workspace ID are available in the environment variables `WINDMILL_TOKEN` and `WINDMILL_WORKSPACE`. The [windmill-cli](https://www.windmill.dev/docs/advanced/cli) (`wmill`) is preinstalled and already configured against this workspace under the local workspace name `evaluation-ws`.

Your job is to build a multi-stage deployment pipeline using Windmill's [Workflows-as-Code](https://www.windmill.dev/docs/core_concepts/workflows_as_code) (WAC) primitives in TypeScript (Bun). The pipeline must run two stages, suspend itself for an external HTTP approval, and only then run a third stage. The verification harness will trigger this workflow via the async webhook, observe the suspended checkpoint, resume it programmatically via the Windmill HTTP API, and verify that the deterministic replay semantics of WAC were respected (no duplicated side effects, no re-execution of completed checkpoints).

## Requirements

1. Read the current evaluation run id from the `ZEALT_RUN_ID` environment variable. Call this value `RUN_ID`.
2. Inside the project at `/home/user/wac-pipeline`, create a workspace folder layout that mirrors a Windmill workspace (i.e. paths under `f/...` relative to the project root). Author the workflow as a Bun TypeScript script at:
   - Workspace path: `f/eval/deploy_pipeline_${RUN_ID}` (no language suffix)
   - On-disk file: `/home/user/wac-pipeline/f/eval/deploy_pipeline_${RUN_ID}.ts`
   - Companion metadata file: `/home/user/wac-pipeline/f/eval/deploy_pipeline_${RUN_ID}.script.yaml` (minimal valid metadata so the `wmill` CLI can deploy it; `bun` language)
3. The workflow must use the `workflow()` wrapper from `windmill-client` and orchestrate, in order, exactly the following checkpoints:
   1. `step("runTests", ...)` — appends the string `"tests-ran"` to an audit array. Inside this step you MUST also atomically increment a Windmill workspace [variable](https://www.windmill.dev/docs/core_concepts/variables_and_secrets) at path `f/eval/runtests_counter_${RUN_ID}` using `wmill.getVariable` + `wmill.setVariable` (the stored value is the integer count as a string).
   2. `step("deployStaging", ...)` — appends `"staging-deployed"`.
   3. `await waitForApproval({ ... })` from `windmill-client` — suspends execution until an external HTTP resume event is received. Use a generous timeout (at least 600 seconds).
   4. `step("deployProduction", ...)` — appends `"production-deployed"`.
   - Finally, the workflow MUST return the audit array directly (a JSON array of three strings in the order above).
4. Deploy the workflow to the Windmill workspace by running the appropriate `wmill` command from the project root so that triggering the workflow's autogenerated async webhook actually executes your code on the cloud workers. The Windmill scripts API path on the deployed instance must end with `f/eval/deploy_pipeline_${RUN_ID}`.
5. Write a short deployment log to `/home/user/wac-pipeline/output.log` containing two lines in the following exact formats (substituting `${RUN_ID}` for the actual run id):
   ```
   Script path: f/eval/deploy_pipeline_${RUN_ID}
   Counter variable: f/eval/runtests_counter_${RUN_ID}
   ```

## Implementation Hints

- Read `RUN_ID` from `ZEALT_RUN_ID` before authoring the file so the run id is hard-coded into the script content, the script path, and the counter variable path.
- The Windmill TypeScript client exposes the WAC primitives directly: `import { workflow, step, waitForApproval } from "windmill-client"`. The same package also exposes `getVariable`/`setVariable`.
- Remember that WAC uses a [checkpoint / replay model](https://www.windmill.dev/docs/core_concepts/workflows_as_code#how-it-works): the parent workflow body is re-invoked when the job resumes. Any code outside a `step()` (or `task()`) will run on every replay; results computed inside a `step()` are persisted and returned from cache on replay. Design the audit / counter logic accordingly.
- The counter variable does not need to exist before the first run — `wmill.getVariable` will throw if missing; catch that case and treat the previous value as `0`. The verification harness will pre-create the variable as `"0"` to keep behaviour deterministic.
- Deployment: with the cloud workspace already registered under the name `evaluation-ws`, `wmill sync push --yes` from the project root, or `wmill script push f/eval/deploy_pipeline_${RUN_ID}`, will publish the script. Make sure the companion `.script.yaml` exists and declares `language: bun`.
- You do NOT need to trigger or resume the workflow yourself — the verification harness does that. You only have to author and deploy the script (and write the log file).

